# Mermaid Conventions

Use Mermaid for **flow, sequence, and state diagrams** — anything with a clear directional process
or set of transitions. Use native shapes + the icon library (see `assets/icons/index.md`) for
**architecture diagrams** — anything about system topology/components rather than process flow.

## When to use which diagram type

| Diagram type | Tool | Mermaid syntax |
|---|---|---|
| Process / activity flow | Mermaid | `flowchart` |
| Sequence of calls between systems/actors | Mermaid | `sequenceDiagram` |
| State machine / lifecycle | Mermaid | `stateDiagram-v2` |
| System/component architecture | Native shapes + icons | n/a — see `solution-diagrams/SKILL.md` |
| Gantt/timeline | Mermaid | `gantt` |

## Theming

Always render with the themed config so Mermaid output matches the active brand palette instead of
Mermaid's default colours. `scripts/render-mermaid.sh` wraps this for you — don't call `mmdc`
directly without it.

The theme config (`references/mermaid-theme.json`, generated by the render script on first run if
missing) maps brand-core palette tokens to Mermaid's theme variables:

```json
{
  "theme": "base",
  "themeVariables": {
    "primaryColor": "#2563EB",
    "primaryTextColor": "#FFFFFF",
    "primaryBorderColor": "#1E293B",
    "lineColor": "#64748B",
    "secondaryColor": "#F59E0B",
    "tertiaryColor": "#10B981",
    "fontFamily": "Arial, sans-serif"
  }
}
```

Update the hex values to match your real brand-core palette when you populate the private overlay.

## Flowchart Conventions

- Direction: `TD` (top-down) for process flows read top-to-bottom; `LR` (left-right) for pipelines/
  stages read as a progression.
- Label every node with a short noun phrase (2–4 words) — avoid full sentences in node labels.
- Use subgraphs to group nodes by phase/system/owner when a flow spans more than one boundary:
  ```
  flowchart LR
    subgraph Source Systems
      A[CRM] --> B[Data Export]
    end
    subgraph Platform
      B --> C[Ingestion] --> D[Processing]
    end
  ```
- Decision nodes use the diamond shape (`{Decision?}`) and every outgoing edge is labelled
  (`-->|Yes|` / `-->|No|`) — never leave a branch unlabelled.

## Sequence Diagram Conventions

- Order participants left-to-right in the order the interaction actually starts (usually: end
  user/client system first, then the systems it calls, in call order).
- Use `activate`/`deactivate` to show processing time on the receiving side of any call that isn't
  instantaneous.
- Prefer solid arrows (`->>`) for requests and dashed arrows (`-->>`) for responses so the
  request/response pairing is visually obvious without reading every label.

## State Diagram Conventions

- Always include a `[*]` start state and at least one terminal state.
- Label transitions with the triggering event, not just the destination state.

## Common Gotcha

Any literal `&` in a Mermaid label will render incorrectly or break parsing in some renderers —
use `and` or escape as `&amp;` depending on the render path. Check rendered output before embedding
in a final deliverable.

## Rendering

Always render through `scripts/render-mermaid.sh <input.mmd> <output.svg|png>` — see that script for
usage. Never hand-embed raw Mermaid syntax into a deck or document; always render to SVG/PNG first
and embed the image, since not every viewer renders Mermaid natively.

**Embedding in a PowerPoint deck — always render to PNG, not SVG.** Common deck-building libraries
(e.g. pptxgenjs) do not properly rasterize SVG on `addImage()` — they can silently write a file with
a `.png` extension that's actually unconverted SVG bytes, which renders blank in any viewer that
doesn't specifically handle the DrawingML SVG extension (older PowerPoint, LibreOffice, Keynote, most
converters). Render Mermaid output to `.png` for any deck embed. SVG output is safe for Word/document
embeds and for further vector editing, but never for a deck.
