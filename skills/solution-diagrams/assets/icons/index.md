# Icon Index

Check this index before drawing any architecture diagram by hand. If an icon exists here for a
system/concept you need, use it — don't improvise a new shape for something already covered.

**This is a placeholder index.** Populate it with your firm's real icon set (product logos,
platform icons, generic concept icons) via the private overlay at
`private/skills/solution-diagrams/assets/icons/`, mirroring this folder structure. Each category
below is a subfolder; update this table to match what you actually have.

| Category | Folder | Contents (examples) |
|---|---|---|
| Cloud platforms | `cloud/` | generic cloud, compute, storage, network icons |
| Data | `data/` | database, data lake, pipeline, warehouse icons |
| Identity & security | `security/` | lock, shield, key, identity-provider icons |
| Integration | `integration/` | API, queue, event-bus, webhook icons |
| Applications | `apps/` | generic app, mobile, web, CRM/ERP-shaped icons |
| People & roles | `people/` | generic user, team, external-party icons |

## Usage Convention

- Icons are simple, single-colour SVGs sized on a square artboard (e.g. 64×64) so they scale
  cleanly at any size without distortion.
- Recolour icons to match the active brand palette (`brand-core`) or the active diagram convention
  (see `references/mermaid-conventions.md`) rather than using icons in their original colour if that
  colour isn't in the palette.
- If no icon exists for a concept you need, use a plain labelled rectangle instead of inventing a
  one-off icon — consistency of "no icon" beats inconsistency of an ad hoc icon style.
- When you add a genuinely reusable icon, add it to the appropriate category folder and update this
  index — don't let the index drift out of sync with what's actually available.
